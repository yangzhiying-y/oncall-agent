# 第一轮优化学习指南：把 Demo 做成可用产品

这份指南配合本次代码一起看。你不需要先懂 Python、网页或 AI；先知道
“每个文件负责什么”，再一点点看代码就够了。

## 界面重做补充：为什么新页面要单独建文件

旧首页的样式积累太多，规则互相覆盖，才会出现图标突然变得巨大、文字颜色失控这类问题。修补一两条 CSS 只能掩盖问题，不能让页面重新变得可靠。

现在首页使用 `static/chat.html` 和 `static/chatgpt.css`：

- `chat.html` 只保留 GPT 式聊天界面真正需要的结构；
- `chatgpt.css` 是独立样式，不再与旧的 `styles.css` 互相影响；
- 左边只做导航，正中只做聊天，底部只做输入；
- 旧 `index.html` 会自动跳转到新版，因此正在运行的服务刷新后也能切换。

这叫做**隔离式重构**：当旧代码已经很乱时，先让新实现独立稳定，再逐步删除旧实现。比在旧样式里不断加 `!important` 更容易维护。

## 1. 先认识这套项目

把项目想成一家小餐厅：

| 位置 | 它负责什么 | 你通常什么时候修改它 |
| --- | --- | --- |
| `static/index.html` | 页面骨架：按钮、输入框、文字 | 想增加一个可见功能时 |
| `static/styles.css` | 页面外观：颜色、间距、手机适配 | 想让页面更好看、更易用时 |
| `static/app.js` | 页面行为：点击、请求接口、显示结果 | 想让按钮真的“会做事”时 |
| `app/main.py` | FastAPI 总入口和浏览器安全规则 | 新增全局能力时 |
| `app/api/file.py` | 文档上传接口 | 调整文件类型、大小或上传流程时 |
| `app/config.py` | 从 `.env` 读取配置 | 新增可配置项时 |

浏览器负责显示界面；`app` 目录里的 Python 负责处理问题；Milvus 负责记住
文档内容。三者连起来，才是一个完整的 Agent 应用。

## 2. 本轮做了哪些改动

### 2.1 页面不再像一张空白聊天页

`index.html` 新增了三个区域：

1. 左侧品牌与会话区域：新建对话、导入知识文档、历史记录。
2. 顶部工作台区域：告诉使用者当前在什么产品里，并显示服务状态。
3. 欢迎区：把“知识问答、故障诊断、文档入库”三种能力直接说清楚，并给出可点击的提问示例。

一个好界面的第一原则是：**不要让用户猜下一步做什么。**

例如，新增一个按钮的最小写法是：

```html
<button type="button" data-prompt="如何排查服务响应缓慢？">
  如何排查服务响应缓慢？
</button>
```

`data-prompt` 是给 JavaScript 看的“小标签”。它不会显示给用户，却能让程序
知道：点击按钮后，要把哪句话放进输入框。

### 2.2 CSS 先定义设计变量，再写具体样式

`styles.css` 末尾新增了这一类代码：

```css
:root {
  --brand: #0f766e;
  --ink: #15221f;
  --canvas: #f5f8f6;
}
```

这里的 `--brand` 可以理解为“品牌主色的昵称”。以后你想把绿色整体改成蓝色，
不用全文件搜索颜色，只改一行即可：

```css
--brand: #2563eb;
```

本轮还给按钮加入了键盘焦点样式、给手机屏幕设置了响应式规则。它们看起来不抢眼，
但能让键盘用户和手机用户正常完成操作。

### 2.3 快捷问题和服务状态是如何工作的

在 `app.js` 的 `bindEvents()` 中，每个示例问题都会注册一次点击事件：

```javascript
button.addEventListener('click', () => {
  this.messageInput.value = button.dataset.prompt || '';
  this.messageInput.focus();
});
```

读法很朴素：

- `button`：当前被点击的按钮；
- `dataset.prompt`：HTML 中的 `data-prompt` 值；
- `value`：输入框里的文字；
- `focus()`：把光标放进输入框，让用户确认或修改后再发送。

服务状态则会调用 `/health`。绿色代表应用和 Milvus 都可用；红色代表需要检查
服务。静态页面预览时显示“暂不可用”是正常的，因为那种预览没有启动后端。

### 2.4 为什么要专门处理 Markdown 安全

AI 回答和上传文档都属于**不可信输入**。以前的代码把 Markdown 转成 HTML 后
直接插入页面；如果内容里混进脚本或危险链接，浏览器可能执行它。

现在路径变成：

```text
AI 文本 → marked 转 Markdown → sanitizeHtml 白名单过滤 → 显示在页面
```

`sanitizeHtml()` 只允许标题、段落、列表、代码块、表格和安全链接等常用格式；
脚本、内嵌页面、图片和 `javascript:` 链接会被移除。以后如果你想支持一种新标签，
不要直接放开全部 HTML，而是把它谨慎加入 `allowedTags`。

### 2.5 上传文件为什么改成“先检查，后覆盖”

旧顺序是：先删除旧文件，再读取和检查新文件。若新文件过大或为空，原文件已经没了。

新顺序是：

```text
读取新文件 → 检查是否为空、是否超过 20 MB → 写入覆盖 → 建立向量索引 → 返回结果
```

这是一条很实用的程序设计原则：**先验证，再修改旧数据。**

前端和后端现在都限制为 20 MB，也都支持 `.markdown`。如果文件保存成功但向量索引
失败，页面会明确告诉你“暂时不能检索”，而不是假装一切正常。

### 2.6 `.env` 是什么，为什么不能上传到 Git

`.env` 是本机私密配置，尤其可能包含 API Key。现在项目提供了
`.env.example`：它只放字段名和示例值，可以放心分享；真正的 `.env` 已加入
`.gitignore`，不会被 Git 默认跟踪。

第一次配置时：

```powershell
Copy-Item .env.example .env
notepad .env
```

然后把 `DASHSCOPE_API_KEY=replace-with-your-api-key` 改成自己的真实 Key。
真实 Key 不要发到聊天记录、截图、仓库或作业附件里。

### 2.7 为什么启动时不再自动重复导入示例文档

`start-windows.bat` 现在会询问是否重新导入 `aiops-docs`。第一次启动或你修改了这些示例文档时选 `Y`；日常启动选 `N`。这样可以避免每次启动都重复建索引，既节省模型调用，也减少知识库出现重复片段的风险。

## 3. 你以后自己加一个功能，按这个顺序做

以“增加一个清空全部本地历史的按钮”为例：

1. 在 `index.html` 放一个有唯一 `id` 的按钮。
2. 在 `styles.css` 给它设定正常、悬停、键盘聚焦三种状态。
3. 在 `app.js` 的 `initializeElements()` 找到这个按钮。
4. 在 `bindEvents()` 里给它添加点击事件。
5. 编写一个小方法完成业务动作；操作 `localStorage` 前，先确认不会误删。
6. 打开浏览器，亲手点一次；再刷新页面确认结果。

这六步比“直接把一大段代码粘进去”慢一点，却更不容易出错。每当你卡住，先问自己：
“这件事是页面结构、页面样式、页面行为，还是后端数据？”答案通常会告诉你该打开哪个文件。

## 5. 知识库引用不是让模型“自己编”

当 Agent 使用知识库检索工具时，回答末尾会自动附上实际检索到的文件名和标题层级。这不是让模型凭空写引用，而是从工具返回的文档元数据中提取，因此更可信。
